# 第十七章：形式驗證

## 17.1 形式驗證的動機

形式驗證（Formal Verification）是指使用數學方法來證明軟件或硬件系統的正確性。與傳統的測試方法不同，形式驗證試圖通過嚴格的數學推理來確保系統沒有錯誤。

為什麼需要形式驗證？傳統的軟件開發方法（如測試）只能發現錯誤的存在，但不能證明錯誤的不存在。即使經過數百萬次測試，仍可能存在未被發現的邊界情況錯誤。

在一些安全關鍵系統中——如航空電子設備、核電站控制系統、醫療器械——錯誤可能導致災難性的後果。形式驗證提供了一種方式來確保這些系統的正確性，達到傳統測試無法達到的保證水平。

形式驗證的歷史可以追溯到計算機科學的早期。1970年代，羅伯特·弗洛伊德（Robert Floyd）和托尼·霍爾（C. A. R. Hoare）獨立地發展了程序驗證的公理化方法。此後，形式驗證經歷了漫長的發展，現在已經成為一個成熟的領域。

## 17.2 霍爾邏輯

霍爾邏輯（Hoare Logic）是由托尼·霍爾（C. A. R. Hoare）於1969年提出的一種用於驗證程序正確性的形式系統。霍爾邏輯使用「霍爾三元組」來描述程序的語義：

{P} C {Q}

其中：
- P 是前置條件（precondition），描述程序執行前的狀態
- C 是程序或命令
- Q 是後置條件（postcondition），描述程序執行後的狀態

霍爾三元組 {P} C {Q} 的意思是：如果程序 C 執行前 P 為真，並且程序 C 會終止，那麼執行後 Q 為真。

霍爾邏輯為不同的程序結構提供了推論規則：

**賦值規則**：
{P[E/x]} x := E {P}
（要證明 {P} x := E {Q}，只需將 Q 中的 x 替換為 E 得到 P）

**複合規則**：
如果 {P} C1 {R} 且 {R} C2 {Q}，則 {P} C1; C2 {Q}

**條件規則**：
如果 {P∧B} C1 {Q} 且 {P∧¬B} C2 {Q}，則 {P} if B then C1 else C2 {Q}

**循環不變量**：
如果 {P∧B} C {P}，則 {P} while B do C {P∧¬B}

霍爾邏輯是程序驗證的基石。雖然它不能處理並發程序，但它為後續的更強大方法奠定了基礎。

## 17.3 モデル檢查

モデル檢查（Model Checking）是一種自動化的形式驗證技術。在模型檢查中，我們：
1. 將系統建模為有限狀態機（如 Kripke 結構）
2. 將規範表示為時態邏輯公式
3. 使用算法系統性地檢查系統是否滿足規範

**時態邏輯**：時態邏輯允許我們表達時間概念。常用的時態邏輯包括：
- LTL（Linear Temporal Logic）：描述系統執行路徑上的性質
- CTL（Computation Tree Logic）：描述系統執行樹上的性質

例如，「每次啟動後，最終會到達關機狀態」可以用 LTL 寫成：□(start → ○□shutdown)

**符號模型檢查**：1981年，布萊克特（Ken McMillan）發明了符號模型檢查，它使用布爾公式（尤其是 OBDD）來表示狀態集合。這使得模型檢查能夠處理非常大的狀態空間。

**有界模型檢查**：有界模型檢查將問題歸約為布爾可滿足性（SAT）問題，可以使用現代 SAT 求解器有效地解決。

## 17.4 Coq 與構造證明助手

Coq 是一個基於類型理論的證明助手，它允許用戶編寫數學證明並由計算機驗證。Coq 使用 Calculus of Inductive Constructions（CiC）作為其基礎邏輯，這是一種極為強大的依賴類型理論。

Coq 的工作流程如下：
1. 用戶陳述一個命題
2. 用戶提供一個「證明腳本」
3. Coq 驗證腳本是否確實證明了命題

Coq 已被用於驗證許多重要的數學定理和軟件：

**四色定理**：1996年，Coq 被用來驗證四色定理的證明。這是首次主要數學定理的計算機輔助證明被形式化驗證。

**CompCert**：CompCert 是一個用 Coq 驗證的 C 編譯器。它證明了編譯器不會引入錯誤——源 C 程序與編譯後的程序在行為上是等價的。這對於安全關鍵系統非常重要。

**seL4 微內核**：seL4 是一個已驗證的安全微內核。它的正確性已經用 Isabelle/HOL（另一個證明助手）證明。

**F* 語言**：F* 是一種基於依賴類型的函數式語言，它同時是一種程序設計語言和一個證明助手。

## 17.5 抽象解釋

抽象解釋（Abstract Interpretation）是一種用於近似分析程序語義的理論框架。它由帕特里克· Cousot 和 Radhia Cousot 於 1977 年提出。

抽象解釋的基本思想是：
- 程序的「具體語義」是精確的，但通常無法直接計算
- 我們可以定義一個「抽象語義」，它是具體語義的近似
- 抽象語義可以在抽象域上計算，提供有用的信息

**抽象域**：抽象解釋使用「抽象域」來表示程序狀態的近似。例如：
- 符號域：追蹤變量的符號（正、負、零）
- 區間域：追蹤變量的取值範圍
- 凸多邊形域：追蹤變量之間的線性關係

**應用**：抽象解釋已被用於：
- 編譯器優化
- 程序分析
- 錯誤檢測

例如，著名的靜態分析工具 Astrée 就是基於抽象解釋的，它已被用於驗證航空電子軟件。

## 17.6 形式驗證的實際應用

形式驗證已經在實際系統中得到了應用。

**航空電子**：NASA 和 ESA 廣泛使用形式方法來驗證航空電子軟件。例如，IBM 的嘗試（the IBM attempt）項目使用模型檢查來驗證飛機控制軟件。

**芯片設計**：英特爾、AMD 等芯片製造商使用形式驗證來確保芯片設計的正確性。處理器中的「bug」代價極高，因此晶片公司在驗證上投入巨資。

**加密協議**：加密協議的形式驗證可以發現微妙的安全漏洞。Tamarin、ProVerif 等工具被用於分析 SSL/TLS、WireGuard 等協議。

**區塊鏈**：智能合約的正確性至關重要——它們直接處理金錢。形式驗證被用來確保智能合約不會被利用。

## 17.7 形式驗證的挑戰與未來

形式驗證面臨著許多挑戰：

**擴展性**：形式驗證通常只適用於相對較小的系統。對於大型系統，必須使用抽象和分解技術。

**自動化程度**：某些驗證任務仍然需要大量的人工干預。提高自動化程度是一個重要的研究方向。

**可用性**：形式驗證工具需要高水平的專業知識才能使用。改進可用性可以使形式驗證更廣泛地採用。

**與傳統工程的集成**：將形式驗證整合到現有的軟件開發流程中是一個挑戰。

**AI 輔助驗證**：近年來，使用機器學習來輔助形式驗證的研究越來越多。例如，使用神經網絡來引導定理證明器的搜索。

## 17.8 小結

本章我們介紹了形式驗證，這是計算理論在確保系統正確性方面的實際應用。我們討論了霍爾邏輯、模型檢查、Coq 證明助手、抽象解釋等核心技術，以及形式驗證在航空電子、芯片設計、加密協議等領域的應用。

形式驗證代表了計算理論與軟件工程的深層結合。通過使用數學證明來確保系統的正確性，我們可以達到傳統測試方法無法比擬的保證水平。雖然形式驗證仍有挑戰要克服，但它正在變得越來越重要，特別是在安全關鍵系統中。
