# 9. 高階作業系統

作業系統是現代計算機系統的核心軟體，負責管理硬體資源並為應用程式提供抽象化的介面。桌上型與筆記型電腦所使用的高階作業系統，如 Windows、macOS 和 Linux，需同時滿足多種需求：高效能、多工處理、多媒體支援、網路功能、安全性，以及豐富的使用者介面。本章將深入探討 Windows NT 和 macOS XNU 兩大主流桌面作業系統的核心架構設計，涵蓋程序管理、記憶體管理、檔案系統、安全機制等核心子系統的理論基礎與實作細節。

## 9.1 Windows 核心結構

### 9.1.1 Windows NT 核心架構

Windows NT（New Technology）是由微軟開發的作業系統家族，最初由 Dave Cutler 和 Mark Lucovsky 在 1993 年設計。NT 家族的設計理念是建立一個高度可移植性、安全可靠且支援多元運算的作業系統核心。與傳統的 MS-DOS 和 Windows 9x 家族不同，Windows NT 從一開始就採用了更現代化的系統架構。

Windows NT 的核心架構採用所謂的「混合核心」（Hybrid Kernel）設計，這是一種介於傳統單核心（Monolithic Kernel）和微核心（Microkernel）之間的架構選擇。純粹的微核心設計將大多數作業系統服務移到使用者空間，核心只保留最基本的功能，如行程排程和低階 IPC。這種設計在理論上具有良好的可擴展性和穩定性，但頻繁的核心-使用者空間切換會帶來顯著的性能開銷。Windows NT 的混合設計則在核心空間中包含了更多的系統服務，以減少上下文切換的頻率，同時保持良好的模組化結構。

**核心架構層次**

```
┌─────────────────────────────────────────────┐
│              使用者模式 (User Mode)            │
│  應用程式、Win32 API、子系統               │
├─────────────────────────────────────────────┤
│              系統服務層                       │
│  Executive（I/O、物件管理、程序管理）        │
├─────────────────────────────────────────────┤
│              核心層 (NT Kernel)              │
│  排程、異常處理、同步                      │
├─────────────────────────────────────────────┤
│              硬體抽象層 (HAL)               │
└─────────────────────────────────────────────┘
```

在這個架構中，**使用者模式**（User Mode）包含所有應用程式以及提供 API 介面的子系統函式庫。Windows 支援多種應用程式子系統，包括 Win32、POSIX 和 OS/2，其中 Win32 是最主要且功能完整的子系統。**Executive** 是作業系統的主要服務層，實作了 I/O 管理、物件管理、程序管理、虛擬記憶體管理、安全性等核心功能。Executive 運行在提升的特權模式下，能夠直接存取硬體資源。**NT 核心**（通常稱為「微核心」）是整個系統的核心，提供最基本、最頻繁使用的作業系統原語，包括行程與執行緒的排程、例外與中斷處理、低階同步機制等。**硬體抽象層**（HAL）則將作業系統與硬體平台的差異抽象化，使得上層軟體無需直接處理特定硬體的細節。

值得強調的是，NT 核心與 Executive 的區分是理解 Windows 架構的關鍵。NT 核心（ntoskrnl.exe 中的内核部分）提供了低階的排程和同步原語，其程式碼被設計為盡可能精簡和穩定。Executive（ntoskrnl.exe 中的其餘部分）則實作了大多數作業系統服務，如記憶體管理、程序管理、I/O 操作等。當應用程式呼叫系統服務時，控制流從使用者模式進入 Executive，而非直接進入核心，這種設計簡化了系統服務的實作。

### 9.1.2 Executive 服務

Executive 是 Windows NT 作業系統的核心服務層，實作了多個相互協作的子系統，每個子系統負責特定領域的功能。

**物件管理員**（Object Manager）是 Windows 作業系統物件導向管理的核心元件。在 Windows 中，几乎所有系統資源——包括行程、執行緒、檔案、事件、mutex、共享記憶體等——都以物件的形式呈現。物件管理員維護一個層級化的命名空間，為每個物件分配安全描述符，並管理物件的生命週期。當應用程式請求建立或開啟一個物件時，物件管理員負責驗證權限、在命名空間中定位或建立物件、並返回一個句柄（handle）給應用程式。這種統一的物件模型簡化了系統資源的管理，同時為安全性檢查提供了集中化的機制。

**程序管理員**（Process Manager）負責行程和執行緒的建立、終止和狀態追蹤。在 Windows 中，行程是資源分配的單位，而執行緒是排程和執行的單位。每個行程擁有一個或多多個執行緒，每個執行緒共享其父行程的位址空間和資源。程序管理員維護行程和執行緒的物件結構，追蹤其狀態（就緒、執行、等待、終止等），並與排程器協作以實現公平和高效 CPU 時間分配。

**虛擬記憶體管理員**（Memory Manager）實作了 Windows 的虛擬記憶體系統。32 位元 Windows 典型配置提供 2GB 用戶位址空間和 2GB 核心位址空間；而 64 位元 Windows 則支援巨大的虛擬位址空間，理論上可達 16EB（16 exabytes），但實際使用會受到硬體和作業系統政策的限制。記憶體管理員負責虛擬位址到實體位址的轉換、頁面置換、工作集管理、記憶體對應檔案等功能。它使用分頁級別的保護機制，確保行程之間的記憶體隔離。

**I/O 管理員**（I/O Manager）為各種 I/O 設備提供統一且一致的介面。它維護設備驅動程式的層級結構，支援同步和非同步 I/O 操作，並實現了 Plug and Play（PnP）和電源管理的支援。I/O 管理員定義了一套標準的驅動程式模型，使得新硬體的支持可以通過開發對應的驅動程式來添加，而無需修改作業系統的核心。

| Executive 服務 | 核心功能 |
|---------------|----------|
| Object Manager | 統一物件命名空間、資源追蹤 |
| Process Manager | 行程/執行緒生命週期管理 |
| Memory Manager | 虛擬記憶體、頁面置換、工作集 |
| I/O Manager | 統一 I/O 介面、驅動程式管理 |
| Security Monitor | 存取控制、安全描述符 |
| LPC Facility | 本地程序呼叫（輕量 IPC） |

**本地程序呼叫**（Local Procedure Call, LPC）是 Windows  Executive 內部的程序間通訊機制，用於同一機器上不同行程之間的快速通訊。LPC 採用客戶端-伺服器模型，客戶端行程透過 LPC 將請求傳送給服務行程，後者處理請求並返回結果。Win32 子系統的許多 API 呼叫實際上透過 LPC 與 CSRSS（Client/Server Runtime Subsystem）行程通訊。

### 9.1.3 系統呼叫介面

在傳統 Unix 系統中，應用程式透過明確的系統呼叫（system call）介面與核心通訊。Windows NT 的設計理念有所不同——應用程式主要透過 API 函式庫（如 kernel32.dll、advapi32.dll、user32.dll、gdi32.dll）與系統互動，而非直接呼叫系統呼叫。這些 API 函式庫在內部負責將高階 API 呼叫轉換為適當的系統服務請求。

然而，了解 Windows 的系統服務呼叫機制對於系統程式設計和偵錯仍然非常重要。Windows 使用 `sysenter`（在較舊的處理器上）或 `syscall`（在較新的 x86-64 處理器上）指令觸發從使用者模式到核心模式的轉換。系統服務分派表（System Service Descriptor Table, SSDT）將服務編號映射到對應的核心函式。Windows 的系統服務編號前綴（如 `NtCreateFile`、`NtReadFile`、`NtWriteFile` 等）反映了與 NT Executive 服務的直接對應關係。

### 9.1.4 Win32 API 程式設計

Win32 API 是 Windows 應用程式開發的基礎，提供了對作業系統功能的完整存取。以下範例展示了一些常見的 Win32 API 使用模式：

```c
// Windows API 程式設計範例
int main() {
    // 建立程序
    STARTUPINFO si = { sizeof(si) };
    PROCESS_INFORMATION pi;
    CreateProcess("notepad.exe", NULL, NULL, NULL, FALSE,
                 0, NULL, NULL, &si, &pi);
    
    // 等待程序結束
    WaitForSingleObject(pi.hProcess, INFINITE);
    
    // 關閉句柄
    CloseHandle(pi.hProcess);
    CloseHandle(pi.hThread);
    
    // 記憶體配置
    LPVOID mem = VirtualAlloc(NULL, 4096,
                             MEM_COMMIT, PAGE_READWRITE);
    
    // 寫入記憶體
    memcpy(mem, "Hello", 5);
    
    // 釋放記憶體
    VirtualFree(mem, 0, MEM_RELEASE);
    
    // 檔案操作
    HANDLE file = CreateFile("test.txt", GENERIC_WRITE, 0,
                              NULL, CREATE_ALWAYS, FILE_ATTRIBUTE_NORMAL, NULL);
    if (file != INVALID_HANDLE_VALUE) {
        DWORD written;
        WriteFile(file, "Hello Windows!", 13, &written, NULL);
        CloseHandle(file);
    }
    
    // 建立執行緒
    HANDLE thread = CreateThread(NULL, 0, threadFunc, NULL, 0, NULL);
    WaitForSingleObject(thread, INFINITE);
    CloseHandle(thread);
    
    return 0;
}

DWORD WINAPI threadFunc(LPVOID param) {
    printf("Thread running\n");
    return 0;
}
```

Win32 API 的設計遵循了物件導向的原則，儘管它使用 C 語言實作。幾乎所有可管理的系統資源都透過句柄（handle）來引用。句柄是一個不透明的值，由系統核心分配和管理；應用程式只需將其作為參數傳遞給相關 API 呼叫，而無需了解其內部結構。當不再需要某個資源時，應該呼叫對應的 CloseHandle 函式來釋放資源，避免資源洩漏。

## 9.2 macOS 與 XNU 核心

### 9.2.1 XNU 核心架構

XNU（X is Not Unix）是 Apple 公司為 macOS（先前為 Mac OS X）和 iOS 作業系統開發的核心。它起源於 NeXT 公司的 OPENSTEP 作業系統，後來與 Mach 微核心和 BSD 整合，形成今日的混合核心架構。XNU 的設計目標是結合多個成熟作業系統設計的優點，提供高效能、穩定可靠且支援現代運算需求的系統核心。

XNU 的架構可以分為三個主要層次，每個層次實作特定的功能：

```
┌─────────────────────────────────────────────┐
│              IOKit                           │
│  物件導向驅動框架、電源管理                  │
├─────────────────────────────────────────────┤
│              BSD Layer                       │
│  程序管理、檔案系統、POSIX                   │
├─────────────────────────────────────────────┤
│              Mach Layer                      │
│  微核心：IPC、虛擬記憶體、任務              │
└─────────────────────────────────────────────┘
```

**Mach 層**是 XNU 的核心基礎，提供了最基本的作業系統功能。Mach 的設計哲學強調清晰的介面定義和高度的模組化。Mach 最初是卡內基美隆大學開發的研究專案，旨在探索微核心架構的實際可行性。Mach 引入的許多概念——如訊息傳遞 IPC、虛擬記憶體物件、任務和執行緒的區分——對後續作業系統設計产生了深遠的影響。

**BSD 層**建立在 Mach 層之上，提供了 Unix 相容的程式設計介面和作業系統服務。BSD 層負責實作 POSIX 標準定義的 API，包括程序管理（fork、exec、pthread）、檔案系統操作、網路堆疊的一部分等。BSD 層透過使用 Mach 層提供的基礎設施來實作這些高階功能，同時隱藏了 Mach 的某些複雜性。

**IOKit**是一個物件導向的驅動程式框架，專為 Apple 平台設計。IOKit 借鑒了 C++ 的設計思想，但使用 C 語言的子集來實作，以避免在核心空間使用完整的 C++ 標準庫。IOKit 提供了自動化的電源管理、裝置熱插拔支援、多處理器安全性等特性，大幅簡化了驅動程式的開發。

### 9.2.2 Mach 層特性

Mach 微核心是 XNU 架構的基石，其設計對現代作業系統理論和實作都有重要貢獻。理解 Mach 的核心概念對於理解 macOS/iOS 的運作原理至關重要。

**訊息傳遞**是 Mach 最核心的 IPC 機制。在 Mach 中，所有資源——無論是行程、執行緒、記憶體區域，還是 I/O 設備——都可以透過「連接埠」（port）來引用。連接埠是一個有方向的通訊通道，一端是連接埠的「權利」（port right），另一端是連接埠本身。Mach 的訊息是固定格式的資料區塊，可以包含指標、連接埠權利或其他基本類型的資料。訊息傳遞的一個重要特性是它是「無拷貝」的——訊息的內容實際上並不複製，而是透過記憶體映射和參數傳遞來共享，這大幅提高了大訊息的傳遞效率。

Mach 的虛擬記憶體系統具有高度的靈活性。在 Mach 中，記憶體被建模為「記憶體物件」（memory object），每個虛擬記憶體區域（virtual memory region）與一個記憶體物件關聯。記憶體物件可以有多個「備份儲存」（backing store），包括檔案、設備、或匿名記憶體。當需要讀取某個虛擬記憶體分頁時，Mach 的分頁 Backupd 守護行程負責將資料從備份儲存載入實體記憶體。這種設計支援了記憶體對應檔案、寫時拷貝（copy-on-write）fork、分頁共享等高階功能。

Mach 的任務-執行緒模型將「擁有資源」和「執行指令」的概念分開。**任務**（task）是資源分配的單位，一個任務擁有自己的虛擬位址空間、連接埠空間、以及對各種系統資源的權利。**執行緒**是排程和執行的單位，每個執行緒屬於一個任務，共享該任務的資源。 Mach 的執行緒比傳統 Unix 的行程更加輕量級，因為同一任務中的多個執行緒共享位址空間，切換執行緒不需要切換位址空間。

Mach 還引入了「空中斷」（upcall）機制，這是一種由核心向使用者空間服務的通訊方式。例如，當分頁錯誤發生時，核心可以向使用者空間的分頁 Backupd 發送請求，要求它提供所需的資料。這種設計使得核心保持相對簡單，而複雜的功能可以在使用者空間實作。

| Mach 特性 | 說明 |
|----------|------|
| 訊息傳遞 | 基於連接埠的無拷貝 IPC |
| 虛擬記憶體 | 記憶體物件模型、彈性分頁管理 |
| 任務/執行緒 | 資源與執行單位的分離 |
| 連接埠 | 所有資源的統一抽象 |
| 空中斷 | 核心向使用者空間的請求 |

### 9.2.3 BSD 層特性

BSD 層位於 Mach 層之上，提供了 Unix 標準相容性和豐富的系統 API。BSD 層的實作是 macOS 作為一個 Unix 相容系統的基礎。

**程序管理**在 BSD 層實作。BSD 保留了 Unix 傳統的行程概念，但實際上在內部將其映射到 Mach 的任務-執行緒模型。在 macOS 中，每個 Unix 程序對應一個 Mach 任務，而程序中的第一個執行緒對應 Unix 的主執行緒。BSD 層實作了 `fork`、`vfork`、`exec` 系列、以及 `pthread` 等 POSIX 程序管理函式。值得注意的是，macOS 的 `fork()` 採用了寫時拷貝（copy-on-write）策略，新建立的子行程在初始時與父行程共享所有記憶體分頁，直到任一方嘗試寫入某個分頁時才進行拷貝。這種設計使得 fork 操作變得非常高效，特別適合現代的工作負載。

**檔案系統**方面，macOS 支援多種檔案系統格式。傳統的 HFS+（Hierarchical File System Plus）曾長期是 macOS 的預設檔案系統，但自 macOS High Sierra 起，Apple 引入了 APFS（Apple File System）作為新的預設選擇。APFS 針對固態硬碟（SSD）優化，提供了快照、檔案系統級加密、空間共享等先進功能。BSD 層在 Mach 的虛擬記憶體系統之上實作了 VFS（Virtual File System）層，使得多種檔案系統可以在同一系統中同時使用。

**網路堆疊**方面，macOS 使用了 Berkeley 網路堆疊的實作，這與大多數 Unix 和 Linux 系統相同。BSD 層提供了完整的 TCP/IP 協定棧實作，包括 IPv4、IPv6、TCP、UDP、SCTP 等協定。BSD socket API 是網路程式設計的標準介面，macOS 完全支援。

| BSD 特性 | 說明 |
|---------|------|
| 程序管理 | fork、exec、pthread |
| 檔案系統 | APFS、HFS+、VFS 抽象層 |
| 網路堆疊 | Berkeley TCP/IP 實作 |
| 安全 | MAC 框架、權限模型 |
| 相容性 | POSIX、Unix 標準 |

### 9.2.4 XNU 中的層級整合

理解 XNU 的關鍵在於認識 Mach 和 BSD 層如何協作。雖然這兩個層次在概念上是分開的，但在實際運作中它們緊密整合。

Mach 層維護低階的物件結構，如任務、執行緒、連接埠、記憶體物件等。BSD 層在這些 Mach 物件之上建立更高階的抽象。例如，每個 BSD 程序在內部對應一個 Mach 任務（task），但 BSD 層會在 Mach 任務之上附加額外的結構（如 `proc` 結構），存儲 Unix 程序特有的資訊，如程序 ID、使用者 ID、檔案描述符表等。

Mach 和 BSD 層之間的通訊透過兩種主要機制。首先，BSD 層透過 Mach API 建立和管理 Mach 物件。其次，某些操作會觸發「攔截器」（interceptor）機制。例如，當 BSD 程序呼叫 `fork()` 時，BSD 層會呼叫 Mach 的任務複製功能，然後在新的 Mach 任務基礎上建立 BSD 程序結構。

這種雙層設計的一個實際後果是，某些作業系統行為可以從兩個不同的角度來理解和操作。例如，行程屬性可以透過 BSD 的 `ps` 命令查看，也可以透過 Mach 的 `task_info` API 查詢。這種設計提供了靈活性，但有時也會導致複雜性。

## 9.3 桌面系統的行程與記憶體管理

### 9.3.1 Windows 記憶體管理

Windows NT 的虛擬記憶體管理員（Memory Manager）是 Executive 中最複雜的元件之一，負責管理整個系統的虛擬和實體記憶體資源。它的設計目標是提供：

- **大位址空間**：為應用程式提供遠超過實際實體記憶體的虛擬位址空間
- **記憶體保護**：確保行程之間以及行程與作業系統之間的記憶體隔離
- **記憶體共享**：支援需要共享記憶體的場景
- **透明的頁面置換**：自動將不常用的分頁換出到磁碟，為活動分頁騰出空間

**虛擬位址空間配置**是記憶體管理的基礎。在 32 位元 Windows 中，預設配置是 2GB 用戶位址空間和 2GB 核心位址空間。這種配置可以透過 `/3GB` 開機參數調整為 3GB 用戶空間和 1GB 核心空間。在 64 位元 Windows 中，用戶位址空間理論上可達 16TB（理論極限 16EB，但受到實際記憶體映射限制）。64 位元 Windows 的出現徹底解決了 32 位元系統中的位址空間匱乏問題，使得大型應用程式（如資料庫、影像處理軟體、虛擬化工具）能夠使用巨大的記憶體。

**工作集**（Working Set）是 Windows 記憶體管理的重要概念。它指的是一個行程目前在實體記憶體中的分頁集合。系統會動態調整每個行程的工作集大小，根據行程的記憶體使用模式和系統的整體記憶體壓力情況。工作集管理演算法會嘗試保持「熱」分頁（經常訪問的分頁）在記憶體中，同時將「冷」分頁（很少訪問的分頁）換出到頁面檔案。

Windows 的分頁檔案（pagefile）用於存儲被換出的分頁。當實體記憶體不足時，記憶體管理員會將不活躍的分頁移動到分頁檔案。分頁檔案的大小通常是實體記憶體的 1.5-3 倍，具體取決於系統配置和使用模式。需要注意的是，分頁檔案的過度使用會導致顯著的性能下降，因為硬碟存取速度遠慢於記憶體存取。

**記憶體對應檔案**（Memory-Mapped Files）是 Windows 提供的一種強大機制，允許應用程式將磁碟檔案映射到虛擬位址空間，之後可以像存取記憶體一樣讀寫檔案內容。記憶體對應檔案有以下優點：

- 簡化檔案 I/O：無需显式的 read/write 系統呼叫
- 支援大型檔案：可處理超過虛擬位址空間限制的檔案
- 高效的共享記憶體：多個行程可以映射同一檔案，實現共享記憶體

**虛擬Alloc 系列函式**是 Windows 記憶體配置的基礎 API：

```c
// Windows 記憶體管理範例
void* allocate_and_use() {
    // 保留位址空間（不分配實體記憶體）
    void* base = VirtualAlloc(NULL, 1024 * 1024 * 1024,  // 1GB
                              MEM_RESERVE, PAGE_READWRITE);
    
    // 提交部分記憶體
    void* committed = VirtualAlloc(base, 4096, MEM_COMMIT, PAGE_READWRITE);
    
    // 使用記憶體
    memset(committed, 0x42, 4096);
    
    // 釋放全部記憶體
    VirtualFree(base, 0, MEM_RELEASE);
    
    return NULL;
}
```

VirtualAlloc 函式支援兩階段記憶體分配：先保留（reserve）虛擬位址空間，之後再提交（commit）實際的實體記憶體。這種設計允許應用程式預留大塊位址空間，而只在需要時才消耗實體記憶體資源。

### 9.3.2 macOS 記憶體管理

macOS 的記憶體管理系統結合了 Mach 的虛擬記憶體基礎設施和 BSD 層的 Unix 傳統，同時引入了多項 Apple 特有的創新。

**Mach 虛擬記憶體系統**是 macOS 記憶體管理的基礎。與 Windows 類似，macOS 為每個行程提供獨立的虛擬位址空間。在 64 位元 macOS 上，應用程式可以使用高達 128TB 的虛擬位址空間（理論上 64 位元架構支援 16EB，但 Apple 設定了更實際的限制）。Mach 的分頁大小在 Apple Silicon（M1 及後續）上是 16KB，在 Intel Mac 上是 4KB。

**記憶體壓縮**是 macOS 特有的技術，由 Apple 專利開發。傳統的記憶體管理在實體記憶體不足時會將分頁換出到磁碟（swap），這個過程稱為「分頁輸出」（page out）。分頁輸出雖然能釋放實體記憶體，但磁碟 I/O 的延遲極高，嚴重影響系統回應速度。

記憶體壓縮技術的思路是：當需要釋放記憶體時，先嘗試將記憶體內容壓縮，而不是直接換出到磁碟。macOS 使用專門的壓縮演算法（基於名為「WMZ」的壓縮格式），能夠在極短時間內完成壓縮和解壓縮。如果壓縮後的資料仍然太大，才會進行磁碟換出。

這個機制的優勢在於：

- 速度：壓縮/解壓縮在記憶體中完成，速度比磁碟 I/O 快 10-100 倍
- 節能：避免磁碟spin-up，降低功耗
- 延長 SSD 壽命：減少寫入量
- 應用程式保持回應：交換操作幾乎瞬間完成

**記憶體壓力監控**是 macOS 10.9 引入的功能，為應用程式提供了一種感知系統記憶體壓力的機制。系統維護一個記憶體壓力狀態指標，分為四個等級：

1. **正常**（Normal）：有充足的可用記憶體
2. **警告**（Warning）：可用記憶體開始緊張
3. **危急**（Critical）：記憶體嚴重不足，可能即將觸發殺死行程
4. **昏迷**（Comatose）：極度危急

應用程式可以通過 `libdispatch` 或直接查詢記憶體壓力 API 來監控這個狀態，並主動減少記憶體使用（例如釋放快取、推遲非緊急操作），以避免被系統殺死。

**記憶體保護機制**

macOS 實作了多層次的記憶體保護機制，以防止惡意程式和錯誤程式碼利用記憶體漏洞：

| 機制 | 說明 |
|------|------|
| ASLR | 位址空間配置隨機化，隨機化模組和堆積載入位址 |
| PAC | 指標驗證碼（Pointer Authentication Codes），ARM 特有的硬體保護 |
| Hardened Runtime | 禁用危險的指標操作，加強 JIT 限制 |
| Sandboxing | 應用程式沙盒，限制記憶體映射能力 |

**自動引用計數**（Automatic Reference Counting, ARC）是 Apple 為 Objective-C 和 Swift 引入的記憶體管理機制。不同於 Java 或垃圾回收語言的自動記憶體管理，ARC 在編譯時插入 retain 和 release 呼叫，實現自動化的引用計數管理。這種設計結合了手動記憶體管理的確定性和垃圾回收的安全性的優點，同時避免了垃圾回收帶來的暫停和不可預測性。

### 9.3.3 跨平台記憶體管理考量

在跨平台程式設計中，記憶體管理的差異是需要特別注意的問題。Windows 和 macOS/Linux 在以下方面存在差異：

- **API 差異**：Windows 使用 `VirtualAlloc`、`HeapAlloc`、`GlobalAlloc` 等 API，而 Unix 系統使用 `malloc/free`、`mmap` 等標準 C 函式庫函式
- **堆積策略**：不同系統的預設堆積配置和增長策略可能不同
- **記憶體對應檔案**：雖然兩者都支援，但 API 和行為有細節差異
- **偵錯工具**：Windows 使用較傳統的 CRT 偵錯功能，macOS 使用 `libgmalloc` 等工具

跨平台程式庫（如 Qt、Boost、SDL）通常會抽象這些差異，但在底層系統程式設計中，理解這些差異仍然很重要。

## 9.4 圖形介面與視窗系統

### 9.4.1 X Window System 詳解

X Window System（常簡稱為 X）是 Unix、Linux 以及其他類 Unix 系統上標準的視窗系統。它的設計哲學與其他系統（如 Windows、macOS）有根本性的不同——X 被設計為「網路透明」的客戶端-伺服器架構，這意味著 X 應用程式可以在網路上運行，而顯示在另一台機器上。

**X 的客戶端-伺服器模型**

在 X 的術語中，「X 伺服器」（X Server）是直接管理顯示設備的程式，負責在螢幕上繪製圖形和接收輸入設備（鍵盤、滑鼠）的輸入。「X 客戶端」（X Client）是實際的應用程式，它們請求伺服器執行繪圖操作，而不知道自己正在哪個螢幕上顯示。

```
┌──────────┐      ┌──────────┐      ┌──────────┐
│  客戶端A  │      │  客戶端B  │      │  客戶端C  │
└────┬─────┘      └────┬─────┘      └────┬─────┘
     │                 │                 │
     └────────────────┼─────────────────┘
                      │ 網路（本地或遠端）
                      ▼
              ┌─────────────────┐
              │     X Server     │
              │   (顯示輸出)    │
              └─────────────────┘
```

這種設計帶來了極大的靈活性。例如，使用者可以在筆電上運行一個 X 伺服器（顯示），而 SSH 到遠端伺服器執行圖形應用程式，視窗會在本地的筆電螢幕上顯示。X11（即 X Window System 第 11 版）是目前最廣泛使用的版本。

**X 協定的設計**

X 是一個網路協定，客戶端和伺服器透過網路（或本機連接，如 Unix domain socket）交換訊息。客戶端發送請求（Request）告訴伺服器做什麼，伺服器返回回覆（Reply）和事件（Event）。主要的事件類型包括：

- `Expose`：視窗需要重繪
- `KeyPress`/`KeyRelease`：鍵盤事件
- `ButtonPress`/`ButtonRelease`：滑鼠按鈕事件
- `MotionNotify`：指標移動事件
- `ConfigureNotify`：視窗大小或位置改變

X 的設計遵循「啞終端」模式——伺服器只負責基本的繪圖操作，不理解視窗內容的語義。所有更高階的 UI 元件（如按鈕、選單、滾動條）都需要由客戶端程式庫實作。這與 Windows 和 macOS 的設計形成對比，後者的圖形系統提供了豐富的高階圖形元件。

**X 客戶端函式庫**

大多數 X 應用程式並不直接構造 X 協定訊息，而是使用諸如 Xlib、XCB 或高階工具包（如 GTK、Qt）的客戶端函式庫。Xlib 是最傳統的 C 語言綁定，提供了與 X 伺服器通訊的基礎 API。XCB（X C Binding）是較新的設計，提供了更低層級和更高效能和更高並行性的介面。

```c
// X Window 簡單範例
#include <X11/Xlib.h>
#include <stdio.h>

int main() {
    Display *display = XOpenDisplay(NULL);
    if (!display) {
        fprintf(stderr, "無法開啟顯示\n");
        return 1;
    }
    
    int screen = DefaultScreen(display);
    Window root = RootWindow(display, screen);
    
    // 建立簡單視窗
    Window window = XCreateSimpleWindow(
        display, root,
        100, 100,           // 位置
        400, 300,           // 大小
        1,                  // 邊框寬度
        BlackPixel(display, screen),  // 邊框顏色
        WhitePixel(display, screen)   // 背景顏色
    );
    
    // 選擇感興趣的事件類型
    XSelectInput(display, window, 
                 ExposureMask | KeyPressMask | ButtonPressMask);
    
    // 顯示視窗
    XMapWindow(display, window);
    
    // 事件循環
    XEvent event;
    GC gc = DefaultGC(display, screen);
    
    while (1) {
        XNextEvent(display, &event);
        
        if (event.type == Expose) {
            // 繪製視窗內容
            XDrawString(display, window, gc,
                        50, 50, "Hello X Window!", 14);
        }
        else if (event.type == KeyPress) {
            // 按鍵退出
            break;
        }
    }
    
    XCloseDisplay(display);
    return 0;
}
```

### 9.4.2 Wayland 顯示伺服器

Wayland 是由 KDE 開發者 Kristian Høgsberg 在 2007 年發起的專案，旨在創建一個更簡單、更高效的視窗系統，替代 X Window System。Wayland 的設計反映了過去 20 多年視窗系統發展的經驗教訓。

**Wayland 的設計原則**

Wayland 拋棄了 X 的客戶端-伺服器模型，採用了一種更直接的方式。在 Wayland 中，客戶端（應用程式）直接與顯示器驅動程式和硬體通訊，不需要透過一個獨立的中間伺服器程序。Wayland 協定定義了一套客戶端與合成器（Compositor）之間的通訊標準。

**Wayland 架構**

```
┌─────────────────────────────────────────┐
│           客戶端應用程式                  │
│  (使用 EGL/GLES 渲染到共享緩衝區)        │
└──────────────────┬──────────────────────┘
                   │ Wayland 協定
                   ▼
┌─────────────────────────────────────────┐
│            Wayland 合成器                │
│  (管理輸入、輸出、視窗合成)              │
└──────────────────┬──────────────────────┘
                   │ 直接記憶體存取
                   ▼
┌─────────────────────────────────────────┐
│           DRM/KMS 驅動程式               │
│    (直接渲染管理/核心模式設定)           │
└─────────────────────────────────────────┘
```

Wayland 合成器（如 GNOME 的 Mutter、KDE 的 KWin、Sway 等）負責：

- 管理顯示輸出（解析度、倍率、配置）
- 處理輸入設備事件
- 合成多個客戶端的視窗內容
- 處理視窗生命週期（最大化、最小化、關閉等）

**Wayland 的優勢**

| 特性 | 說明 |
|------|------|
| 直接渲染 | 客戶端直接渲染到視窗，避免多次拷貝 |
| 消除偽影 | 輸入處理在應用程式和合成器之間直接進行，無舊影問題 |
| 簡化協定 | Wayland 協定比 X 協定更簡潔，設計更現代 |
| 更好的觸控支援 | 原生支援多點觸控和手勢 |
| 安全性 | 客戶端無法干擾其他客戶端的輸出 |
| 電源效率 | 減少合成器的複雜性和記憶體拷貝 |

**Weston** 是 Wayland 的參考實作和參考合成器，經常用於嵌入式系統和開發測試環境。**Mutter**（GNOME）和 **KWin**（KDE）是主流桌面環境使用的 Wayland 合成器。**Sway** 是一個用 Wayland 實作的 i3 相容視窗管理器，受到喜歡平鋪式視窗管理的使用者歡迎。

### 9.4.3 桌面環境整合

在 Linux 系統中，視窗系統（X11 或 Wayland）只是整個桌面體驗的一部分。桌面環境（如 GNOME、KDE Plasma、XFCE、LXDE）提供了完整的使用者介面，包括：

- **視窗管理器**：控制視窗的位置、大小、疊放順序、裝飾（標題列、按鈕）等
- **面板/工作區**：提供應用程式啟動器、系統托盤、工作區切換等功能
- **檔案管理器**：提供圖形化的檔案系統瀏覽
- **設定工具**：提供系統配置介面
- **通知系統**：提供系統通知和警示

這些元件與視窗系統緊密整合，共同構成完整的桌面使用體驗。

### 9.4.4 macOS 的視窗系統

相比 Linux 的視窗系統，macOS 採用了一體化的設計。**Quartz** 是 macOS 的圖形核心，包含了多個子系統：

- **Quartz 2D**：2D 圖形渲染引擎，基於 PDF 技術
- **Quartz Extreme**：合成器，負責視窗合成（Intel Mac）或使用 Metal（Apple Silicon）
- **Core Graphics**：底層圖形 API
- **Core Animation**：動畫和轉場效果框架

macOS 的視窗系統整合在 Cocoa 框架中，應用程式透過 `NSWindow`、`NSView` 等類別操作視窗和視圖。與 X11 不同，macOS 不提供網路透明的顯示機制——所有視窗渲染都在本機完成。

## 9.5 桌面 OS 的安全性

### 9.5.1 Windows 安全機制

Windows 作業系統從 Windows Vista 起引入了多層次的安全架構，核心是**使用者帳戶控制**（User Account Control, UAC）機制。

**使用者帳戶控制**（UAC）是一種減少惡意軟體危害的機制。在 Windows Vista 和 Windows 7 中，UAC 以「同意對話框」的形式要求使用者確認管理員權限操作。Windows 8 之後，UAC 支援四種通知级别：

- 始終通知
- 只在應用程式嘗試變更我的電腦時通知
- 只在應用程式嘗試變更電腦時通知（不變暗桌面）
- 永不通知

UAC 的實現依賴於**權限提升**（elevation）機制。當一個需要提升權限的程式被啟動時，系統會創建一個新的、已提升權限的執行個體，並通過 RPC 與原始程序通訊。

**Windows Defender**（在 Windows 10/11 中整合為 Microsoft Defender）是 Windows 內建的防惡意軟體解決方案。它提供了即時保護、離線掃描、網路保護等功能。Microsoft Defender 的核心是一個名為 **反惡意軟體服務**（MsMpEng.exe）的後台服務，它使用基於特徵碼的偵測、行為監控、機器學習等多種技術來識別威脅。

**Windows 防火牆**是 Windows 的有狀態防火牆實現。它監視入站和出站網路流量，根據預定義或自定義的規則決定允許或阻止流量。Windows 防火牆支援以下網路設定檔：

- 網域設定檔（加入 Active Directory 網域時）
- 私人網路設定檔（家庭或工作網路）
- 公用網路設定檔（公共場所）

| Windows 安全機制 | 防護目標 |
|----------------|----------|
| UAC | 防止未授權的系統變更 |
| Windows Defender | 偵測並移除惡意軟體 |
| Windows Firewall | 過濾網路流量 |
| BitLocker | 保護靜態資料 |
| Windows Hello | 無密碼身份驗證 |
| SmartScreen | 防止惡意網站和下載 |

**BitLocker** 是 Windows 的全磁碟加密功能，支援 TPM（可信賴平台模組）和 PIN/密碼兩種解鎖方式。BitLocker 使用 AES 加密保護整個磁碟的內容，即使硬碟被移除並安裝到其他電腦，資料也無法讀取。Windows 10/11 還提供了 **BitLocker To Go**，用於加密 USB 隨身碟等可移除媒體。

### 9.5.2 macOS 安全機制

macOS 的安全架構結合了 Unix 傳統和 Apple 的創新，提供了從硬體到軟體的多層次保護。

**Gatekeeper** 是 macOS 的應用程式驗證機制，確保只有來自可信來源的應用程式才能在系統上運行。Gatekeeper 支援三種設定：

- 只允許來自 App Store 的應用程式
- 允許來自 App Store 和已識別開發者的應用程式
- 允許任何來源的應用程式（等同停用）

已識別開發者的應用程式具有有效的 Apple 開發者 ID 簽章，Gatekeeper 可以驗證這個簽章。當使用者嘗試運行未簽章的應用程式時，系統會發出警告。

**系統完整性保護**（System Integrity Protection, SIP）是 macOS El Capitan 引入的核心安全功能。SIP 限制了 root 使用者的權限，防止即使是 root 用戶也無法修改特定的系統檔案和目錄。這包括：

- `/System` 目錄及其子目錄
- `/usr`（除了 `/usr/local`）
- `/bin`、`/sbin`、`/usr/lib`
- 核心擴充功能（kext）目錄

SIP 透過核心的強制存取控制（MAC）框架實作，為重要的系統資源提供保護。即使攻擊者獲得了 root 權限，也無法繞過這些保護。

**應用程式沙盒**是 macOS 的應用程式隔離機制。沙盒化應用程式只能訪問明確聲明的資源，而無法訪問系統的其他部分。沙盒權限包括：

- 網路存取（用戶端/伺服器/無）
- 檔案存取（特定目錄或僅有使用者選擇的檔案）
- 特定功能（如麥克風、相機、位置服務）
- 使用者選擇的檔案

App Store 上的應用程式必須使用沙盒，否則無法上架。沙盒機制極大地限制了惡意應用程式的破壞範圍——即使一個沙盒化應用程式被完全攻破，攻擊者也無法獲得系統的完整控制權。

**FileVault** 是 macOS 的全磁碟加密功能。使用 XTS-AES-128 加密保護整個啟動磁碟的內容。在 Apple Silicon Mac 上，FileVault 使用專用的安全隔離區（Secure Enclave）處理加密金鑰，提供了硬體級的保護。即使硬碟被物理移除，資料也無法被讀取。

| macOS 安全機制 | 防護目標 |
|---------------|----------|
| Gatekeeper | 驗證應用程式來源 |
| SIP | 保護系統檔案完整性 |
| Sandboxing | 限制應用程式權限 |
| FileVault | 加密磁碟內容 |
| Secure Enclave | 硬體級金鑰管理 |
| Hardened Runtime | 防止程式碼注入和 JIT 操控 |

**Secure Enclave** 是 Apple 設計的硬體安全子系統，存在於 Apple A7 及後續晶片以及 Apple T2 安全晶片中。Secure Enclave 擁有自己的安全開機順序，運行隔離的作業系統（_SEP OS`），無法被主處理器直接存取。它負責處理：

- Face ID 和 Touch ID 的生物特徵比對
- 加密金鑰的生成和儲存
- Apple Pay 交易驗證
- 裝置完整性測量

Secure Enclave 的金鑰使用硬體綁定——即使作業系統被完全攻破，金鑰也無法被提取。這為高敏感操作提供了硬體級的保護。

### 9.5.3 比較分析

Windows 和 macOS 在安全設計上有不同的哲學和側重點：

**Windows 安全特點**：
- 軟體生態系統更為多樣，安全更新速度較慢
- 需要相容大量傳統軟體，某些安全限制較難實施
- UAC 提供了基本的特權提升控制
- BitLocker 整合在企業管理框架中

**macOS 安全特點**：
- 封閉的硬體-軟體生態系統簡化了安全審計
- SIP 從硬體層面限制系統修改
- 沙盒機制在 App Store 應用程式中強制執行
- Secure Enclave 提供硬體級安全隔離

兩者都提供了現代作業系統所需的基本安全功能，但在具體實現和使用者體驗上有顯著差異。選擇哪個平台往往涉及安全性、易用性和生態系統的權衡。

## 9.6 系統效能監控與調校

### 9.6.1 Windows 效能監控工具

Windows 提供了多個內建的效能監控工具：

**工作管理員**（Task Manager）是最常用的系統監控工具，提供程序清單、CPU/記憶體/網路/磁碟使用率的即時顯示。自 Windows 8 起，工作管理員經過重新設計，提供了更清晰的分頁式介面和更豐富的效能資訊。

**資源監控器**（Resource Monitor）提供了更詳細的系統資源使用情況。透過工作管理員的「效能」頁籤或 `resmon.exe` 命令可以啟動。它顯示：

- CPU：每個程序/服務的 CPU 使用率，按核心細分
- 記憶體：每個程序的工作集、提交大小、私有記憶體
- 磁碟：每個程序的磁碟活動（讀取/寫入速度、IO延遲）
- 網路：每個程序的網路傳輸速率

**效能監視器**（Performance Monitor）是 Windows 的進階效能監控工具，支援收集和追蹤效能計數器。它可以用來：

- 長期記錄系統效能指標
- 建立自定義的效能資料收集集合
- 進行效能基準測試

### 9.6.2 macOS 效能監控工具

macOS 提供了類似的監控工具：

**活動監視器**（Activity Monitor）對應 Windows 的工作管理員，顯示所有執行中程序的 CPU、記憶體、能源、磁碟和網路使用情況。選單列工具可以將 CPU 和記憶體使用率常駐顯示。

**`top` 和 `htop`** 命令提供了更靈活的程序監控方式。`htop`（需要另行安裝）提供了彩色的互動式介面，更易於閱讀。

```bash
# macOS 常用的效能監控命令
top -o cpu          # 按 CPU 使用率排序
top -o mem          # 按記憶體使用率排序
vm_stat             # 虛擬記憶體統計
iostat 1            # 每秒磁碟 I/O 統計
netstat -i 1        # 每秒網路介面統計
```

### 9.6.3 效能調校原則

有效的效能調校需要遵循科學的方法：

1. **測量先行**：在修改任何設定之前，先建立效能基準
2. **單一變數**：每次只修改一個變數，以便確定每個變更的影響
3. **監控瓶頸**：使用工具識別系統瓶頸（CPU、記憶體、I/O、網路）
4. **優先順序**：先處理最大的瓶頸，通常會帶來最大的改善
5. **回歸測試**：驗證修改後的系統穩定性和功能正確性

常見的效能瓶頸及其解決方案：

| 瓶頸類型 | 可能的解決方案 |
|---------|---------------|
| CPU 密集 | 程式碼優化、平行化、使用更高效的演算法 |
| 記憶體密集 | 增加記憶體、最佳化記憶體使用、調整分頁檔案 |
| 磁碟 I/O | SSD 升級、磁碟陣列、檔案系統調校 |
| 網路 I/O | 網路優化、本地快取、CDN 使用 |
