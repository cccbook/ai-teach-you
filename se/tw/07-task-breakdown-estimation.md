# 7. 工作分解與估算

## 大任務 = 小步驟

「這個功能需要多長時間？」——這是軟體開發中最難回答的問題之一。AI 可以幫你分解任務、識別依賴，但最終的估算是你的責任。

## 為什麼要分解任務

```
錯誤做法：
「用户登录功能」
↓
预计：2天
↓
实际：2周（因为忘记考虑验证码、第三方登录、找回密码...）

正确做法：
「用户登录功能」拆分为：
├── 前端表单
├── 后端 API
├── 密码验证
├── Session 管理
├── 记住登录
├── 第三方登录（可选）
└── 测试
↓
预计：4天（每个小任务 0.5-1 天）
实际：4天（所有依赖都已考虑）
```

## AI 輔助任務分解

### 基本分解模式

```
你：
請幫我將「電商促銷活動系統」分解為具體的開發任務。

AI：
## 電商促銷活動系統 - 任務分解

### 第一階段：基礎設施
- [ ] 促銷活動資料模型設計
- [ ] 促銷 API 基礎結構
- [ ] 優惠碼生成服務

### 第二階段：核心功能
- [ ] 折扣計算邏輯
- [ ] 訂單應用優惠
- [ ] 庫存預佔/釋放
- [ ] 優惠券領取功能

### 第三階段：規則引擎
- [ ] 促銷規則 DSL 設計
- [ ] 規則引擎實現
- [ ] 規則衝突處理

### 第四階段：展示與用戶端
- [ ] 促銷頁面設計
- [ ] 倒計時顯示
- [ ] 優惠提示組件
- [ ] 用戶優惠券列表

### 第五階段：運營功能
- [ ] 促銷活動管理後台
- [ ] 活動效果報表
- [ ] 庫存預警

### 第六階段：測試與上線
- [ ] 單元測試
- [ ] 集成測試
- [ ] 壓力測試
- [ ] 上線部署
```

### 分解的顆粒度原則

```
太粗（不好）：
[ ] 实现购物车功能

太细（也不好）：
[ ] 实现购物车添加按钮的 CSS 样式
[ ] 实现购物车添加按钮的 hover 效果
[ ] 实现购物车添加按钮的点击事件

合适：
[ ] 购物车添加商品 API
[ ] 购物车页面组件
[ ] 购物车数量更新逻辑
[ ] 购物车删除商品功能
[ ] 购物车数据持久化
```

AI 可以幫你找到這個平衡點。

## 估算技巧

### 樂觀/悲觀/最可能

每個任務給三個估算：

```
任務：用戶認證 API

樂觀估計：1天（一切順利）
最可能估計：2天（正常開發）
悲觀估計：4天（遇到問題）

最終估計 = (樂觀 + 4×最可能 + 悲觀) / 6
        = (1 + 4×2 + 4) / 6
        = 13 / 6
        ≈ 2.2 天
```

讓 AI 幫你計算：

```
你：
帮我估算这个 Sprint 的总工作量：
- 用户认证 API：最可能 2 天，乐观 1 天，悲观 4 天
- 商品列表 API：最可能 3 天，乐观 2 天，悲观 5 天
- 订单创建 API：最可能 4 天，乐观 3 天，悲观 7 天
- 支付集成：最可能 5 天，乐观 4 天，悲观 10 天

AI：
## 估算汇总

| 任务 | 乐观 | 最可能 | 悲观 | 期望值 |
|------|------|--------|------|--------|
| 用户认证 API | 1 | 2 | 4 | 2.2 |
| 商品列表 API | 2 | 3 | 5 | 3.2 |
| 订单创建 API | 3 | 4 | 7 | 4.3 |
| 支付集成 | 4 | 5 | 10 | 5.7 |

总计期望值：15.4 天
Buffer（20%）：3.1 天
总预估：18.5 天 ≈ 3 周（5人天/周）
```

### 識別依賴關係

```
你：
分析这些任务之间的依赖关系：
A. 数据库设计
B. 用户认证 API
C. 商品管理 API
D. 订单 API
E. 支付 API
F. 前端登录页

AI：
## 依赖关系图

A(数据库) ──┬──→ B(用户认证)
           ├──→ C(商品管理)
           ├──→ D(订单)
           └──→ E(支付)

B(用户认证) ──→ F(前端登录页)

D(订单) ──────→ E(支付)
C(商品管理) ──→ D(订单)

## 关键路径
A → B → F（登录页面的关键路径）

## 可并行任务
- A（数据库）可以最先开始
- B、C、D、E 在 A 完成后可以并行开发
- F 在 B 完成后开始
```

## 應對不確定性

### 技術不確定性矩陣

```
                  熟悉度
            低         高
      ┌──────────┬──────────┐
  高  │  探索研究 │   標準開發 │
      │  × 2-3   │    × 1   │
      ├──────────┼──────────┤
  難  │          │          │
 度  │  拆分调研 │  谨慎开发  │
      │  × 3-5   │   × 1.5  │
      └──────────┴──────────┘

× 表示估算倍数
```

### 讓 AI 幫你識別不確定性

```
你：
对于「集成 AI 图像识别自动打标商品」这个功能，
请识别技术不确定性并给出估算建议。

AI：
## 技术不确定性分析

### 高不确定性点
1. AI 模型选型
   - 不确定：哪个模型最适合你的场景
   - 建议：先用 1 周做 PoC

2. 模型部署成本
   - 不确定：QPS 和成本的关系
   - 建议：先评估后端成本

3. 识别准确率
   - 不确定：能否满足业务需求
   - 建议：用户验收测试

### 调整后估算
原始估算：2 周
PoC 阶段：+ 1 周
Buffer（30%不确定性）：+ 0.9 周
调整后估算：3.9 周 ≈ 4 周

### 建议做法
1. 第一周：PoC，验证技术可行性
2. 第二周开始：根据 PoC 结果重新评估
```

## 實際工作流

```
┌─────────────────────────────────────────────────────┐
│              AI 辅助任务分解工作流                    │
├─────────────────────────────────────────────────────┤
│                                                     │
│  1. 需求输入                                         │
│     你：「功能：用户可以分享商品到社交媒体」            │
│                                                     │
│  2. AI 初步分解                                       │
│     AI → 列出所有子任务                              │
│                                                     │
│  3. 你补充遗漏                                       │
│     你：增加「异常处理」「测试」等                    │
│                                                     │
│  4. AI 识别依赖                                       │
│     AI → 生成任务图                                  │
│                                                     │
│  5. 共同估算                                          │
│     你/AI → 每个任务给出三个估算                     │
│                                                     │
│  6. 识别风险                                          │
│     AI → 识别不确定性和风险点                        │
│                                                     │
│  7. 最终决策                                          │
│     你 → 决定最终范围和截止日期                      │
│                                                     │
└─────────────────────────────────────────────────────┘
```
